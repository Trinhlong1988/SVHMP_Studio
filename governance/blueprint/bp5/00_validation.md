# BP5 — 00_validation.md — Validation Architecture (1 cửa)
> Enforce: `tools/blueprint_suite_check.py` (wired stage `blueprint` trong `tools/ci_gate.py`) · chứng thực: `tests/test_bp5_validation.py`.

**Mission:** Hợp nhất bộ checker blueprint (BP0→BP4) thành MỘT cửa máy-kiểm + luật review — từ BP5, mọi commit/push/server-CI tự kiểm toàn bộ blueprint bằng 1 lệnh; BP6-BP8 và G2+ không cần nhớ 5 lệnh riêng.
**Purpose:** Bịt khe hở "checker tồn tại nhưng không gate nào chạy mỗi commit" (named→ENFORCED thật): drift/planned-dishonesty/dup-key ở bất kỳ tầng blueprint nào phải đỏ NGAY tại pre-push + server-CI, không đợi audit cuối pack.
**Scope:** Runner `blueprint_suite_check.py` GỌI tuần tự 5 checker: `blueprint_constitution_check.py` (bp0) · `bp1_architecture_check.py` · `bp2_domain_check.py` · `bp3_ownership_check.py` · `bp4_runtime_check.py` → matrix PASS/FAIL từng tầng, exit 0 chỉ khi TẤT CẢ xanh. KHÔNG viết lại checker con (chống regression-by-overwrite 3/7) — chỉ gọi; sửa checker con = diff tối thiểu + `git log` file đó.
**Authority:** BP0 v2.0 + BP1-BP4 LOCKED = nguồn luật của từng tầng; BP5 không thêm luật nội dung mới — chỉ luật THI HÀNH (khi nào chạy, gộp verdict thế nào). Đổi thành phần suite / gỡ stage = RFC + Mr.Long.
**Responsibilities:** (1) Suite: chạy đủ 5 tầng, không short-circuit (tầng đỏ vẫn chạy tầng sau — matrix đầy đủ cho audit), gộp exit code. (2) DUP-KEY: loader dùng chung `load_yaml_no_dup` — 1 implementation duy nhất trong `blueprint_constitution_check.py`, các checker con IMPORT (không copy-paste 5 bản; khóa bởi `test_dup_key_loader_single_impl`). (3) Wire: stage `('blueprint', 'tools/blueprint_suite_check.py')` trong ci_gate CHECKS → pre-push hook + server-CI (`.github/workflows/ci.yml` chạy ci_gate) cùng enforce. (4) Pass-through `--bp{N} "<args>"` chỉ dành cho negative test/audit trỏ tầng vào file hỏng.
**Workflow (review flow):** Builder sửa/thêm → self-test (`blueprint_suite_check.py` exit 0 + pytest) → commit R200 (worktree riêng, pull --rebase, log_ping, push, KHÔNG --no-verify) → kiểm duyệt audit 7 bước theo `prompts/CMD_AUDIT_PROTOCOL.md` (trỏ — không chép lại) → Mr.Long ký lock/tag. Builder chỉ kết luận READY FOR AUDIT = YES/NO.
**Mandatory Rules:** (1) Suite FAIL bất kỳ tầng → exit 1 → chặn push/CI. (2) Gỡ stage `blueprint` khỏi ci_gate = unwire → `test_blueprint_stage_wired_in_ci_gate` đỏ (mirror test_server_side_ci_wired). (3) DUP-KEY ở bất kỳ file blueprint yaml nào → tầng đó đỏ → suite đỏ. (4) File `planned_path` chạm đất mà chưa flip `planned→exists` (kèm chữ ký Mr.Long vì pack locked) → PLANNED-DRIFT đỏ. (5) Cấm thêm implementation dup-key loader thứ hai trong tools/.
**PASS Criteria:** `blueprint_suite_check.py` exit 0 (matrix 5/5 PASS) + `pytest tests/` xanh (gồm mutation battery test_bp5) — ENFORCED qua ci_gate stage `blueprint`.
**FAIL Criteria:** ≥1 tầng con exit ≠0 → suite exit 1; stage bị gỡ khỏi ci_gate → test unwire đỏ; loader dup-key bị copy-paste bản 2 → test single-impl đỏ.
**Examples:** làm hỏng 1 yaml BP2 (dup-key hoặc facet lạ) → matrix `[FAIL] bp2` + suite exit 1 từ 1 lệnh; xóa dòng stage blueprint trong ci_gate CHECKS → `test_blueprint_stage_wired_in_ci_gate` FAIL; file schemas/ untracked chạm planned_path (sự cố thật 3/7 20:13) → `[FAIL] bp2 PLANNED-DRIFT` ngay lập tức.
**Promotion Rules:** `bp5_validation: candidate` — Builder không lock/tag; Mr.Long ký sau audit 7 bước (reconcile `governance/constitution/00_constitution.md` + `governance/blueprint/00_system_blueprint_constitution.md`).
